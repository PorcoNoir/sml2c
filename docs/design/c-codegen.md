# Compiling SysML to C — design document

This document plans the work after v0.21.  Today's `--emit-c` lowers
*structure* (struct typedefs, top-level static-const declarations,
nested fields, fixed-size arrays).  Behavioral constructs — calc
defs, constraints, actions, state machines — are skipped with
`/* skipped: NAME (not C-emittable) */`.

The pivot: extend `--emit-c` so that a SysML model compiles to a
C library that **runs** the model's behavior, not just describes
its structure.  The clox parallel from *Crafting Interpreters*
Part III: clox compiles Lox to bytecode; we compile SysML to C
function definitions.

---

## 1. The runtime-semantics question

This is the load-bearing decision.  SysML — unlike Lox — doesn't
come with built-in execution semantics.  The OMG spec defines a
structural meta-model and validation rules, and leaves *what
happens when you run a SysML model* to the implementer.

So the question "compile SysML to comparable C code" has a hidden
sub-question: **runnable how?**  Three plausible answers:

### A. Validating library

The C output is a library of typed structs + pure functions.
Calc defs become free functions, defaults become init functions,
constraints become check functions.  No simulation, no time
stepping, no event loop.  The user writes `main()` and calls into
the library to validate or compute.

This is the lightest target.  Closest analog: the C bindings
generated by Protocol Buffers — typed structs + serialize /
deserialize / validate functions, no runtime.

### B. Event-driven simulator

Library-plus: state machines also become enum + step functions,
and actions become sequential function calls.  The user constructs
the model, then calls `step(event)` to advance state.  Each step
fires whichever transitions are enabled and runs whichever actions
they're wired to.  Time is discrete and event-driven.

Closest analog: a hand-written game-loop simulator, or a
ROS-style node that processes one message per tick.

### C. Continuous simulator

Numerical integration over time.  Calc defs evaluated each tick,
state machines transition on guards, actions run continuously.
This is what Simulink and Modelica produce.

C is out of scope — it requires a real ODE solver, a clock model,
and a substantially different lowering for flows.  Mentioned only
to rule it out.

### Decision: A, then B

v0.22–v0.24 target **A** (validating library).  v0.25–v0.27
extend to **B** (event-driven simulator).  Each phase ships
working C and adds one row to the lowering table.

## 2. Running example

The thermostat below is the design's reference model.  Small
enough to fit on one screen, big enough to exercise every
construct in the v0.22–v0.27 lowering table.

```sysml
package Thermostat {
    private import ScalarValues::*;

    /* Pure function: compute output power from temperature error. */
    calc def ResponsePower {
        in error : Real;
        in gain  : Real;
        return out : Real = error * gain;
    }

    /* Three-mode state machine: idle, heating, cooling. */
    state def Mode {
        state idle;
        state heating;
        state cooling;

        transition idle_to_heating  first idle    accept TooCold then heating;
        transition idle_to_cooling  first idle    accept TooHot  then cooling;
        transition heating_to_idle  first heating accept InRange then idle;
        transition cooling_to_idle  first cooling accept InRange then idle;
    }

    action def TooCold;
    action def TooHot;
    action def InRange;

    /* The thermostat's parts and behavior. */
    part def Controller {
        attribute setpoint : Real = 21.0;       // °C target
        attribute measured : Real = 20.0;       // current reading
        attribute deadband : Real = 0.5;        // hysteresis margin
        attribute gain     : Real = 100.0;
        attribute power    : Real = 0.0;        // current output

        assert constraint sane_setpoint {
            setpoint > 10.0 and setpoint < 35.0
        }
        assert constraint sane_deadband {
            deadband > 0.0 and deadband < 5.0
        }

        exhibit state mode : Mode;
    }
}
```

Five things to lower beyond what we already do:

| Construct                       | Lowering target                  |
|---------------------------------|----------------------------------|
| `calc def ResponsePower`        | `double ResponsePower(double, double)` |
| `attribute setpoint = 21.0`     | `Controller_init(self)` body line |
| `assert constraint sane_…`      | branch in `Controller_check(self)` |
| `state def Mode { state idle; …}` | `Mode_state` enum + `Mode_step()` |
| `exhibit state mode : Mode`     | `Mode_state mode` field on Controller |

(The action defs `TooCold` / `TooHot` / `InRange` get used as event
tags; they lower to enum members on `Mode_event`.)

## 3. Generated C — what the user gets

The library splits into a header (declarations, includable from
the user's own code) and an implementation (function bodies).
A `--emit-c-header` flag picks header mode; default is impl.

### `thermostat.h`

```c
/* Generated by sml2c — do not edit. */
#ifndef THERMOSTAT_H
#define THERMOSTAT_H
#include <stdbool.h>
#include <stdint.h>

/* ---- calc defs ---- */
double ResponsePower(double error, double gain);

/* ---- state defs ---- */
typedef enum {
    Mode_idle,
    Mode_heating,
    Mode_cooling
} Mode_state;

typedef enum {
    Mode_event_TooCold,
    Mode_event_TooHot,
    Mode_event_InRange
} Mode_event;

Mode_state Mode_step(Mode_state s, Mode_event e);

/* ---- part defs ---- */
typedef struct {
    double     setpoint;
    double     measured;
    double     deadband;
    double     gain;
    double     power;
    Mode_state mode;
} Controller;

void Controller_init (Controller* self);
bool Controller_check(const Controller* self);

#endif
```

### `thermostat.c`

```c
/* Generated by sml2c — do not edit. */
#include "thermostat.h"

double ResponsePower(double error, double gain) {
    return (error * gain);
}

Mode_state Mode_step(Mode_state s, Mode_event e) {
    switch (s) {
    case Mode_idle:
        if (e == Mode_event_TooCold) return Mode_heating;
        if (e == Mode_event_TooHot)  return Mode_cooling;
        return s;
    case Mode_heating:
        if (e == Mode_event_InRange) return Mode_idle;
        return s;
    case Mode_cooling:
        if (e == Mode_event_InRange) return Mode_idle;
        return s;
    }
    return s;  /* unreachable but keeps -Wreturn-type happy */
}

void Controller_init(Controller* self) {
    self->setpoint = 21.0;
    self->measured = 20.0;
    self->deadband = 0.5;
    self->gain     = 100.0;
    self->power    = 0.0;
    self->mode     = Mode_idle;
}

bool Controller_check(const Controller* self) {
    /* sane_setpoint */
    if (!((self->setpoint > 10.0) && (self->setpoint < 35.0))) return false;
    /* sane_deadband */
    if (!((self->deadband > 0.0) && (self->deadband < 5.0)))   return false;
    return true;
}
```

### A user's `main.c` that uses it

```c
#include "thermostat.h"
#include <stdio.h>

int main(void) {
    Controller c;
    Controller_init(&c);

    if (!Controller_check(&c)) {
        fprintf(stderr, "constraint violation in defaults\n");
        return 1;
    }

    /* Read sensor — for demo, hard-coded. */
    c.measured = 18.0;

    /* Decide which event to inject. */
    double err = c.setpoint - c.measured;
    Mode_event e;
    if      (err >  c.deadband) e = Mode_event_TooCold;
    else if (err < -c.deadband) e = Mode_event_TooHot;
    else                        e = Mode_event_InRange;

    c.mode  = Mode_step(c.mode, e);
    c.power = ResponsePower(err, c.gain);

    printf("mode=%d power=%.2f\n", c.mode, c.power);
    return 0;
}
```

This compiles with `cc -std=c11 main.c thermostat.c -o thermostat`.
It is real, testable, runnable C.  The user's `main.c` is the only
hand-written piece — everything else comes from the SysML.

## 4. The lowering table (full v0.22+)

| SysML construct                              | C lowering                                         | Phase  | Notes                                                |
|----------------------------------------------|----------------------------------------------------|--------|------------------------------------------------------|
| `attribute x : T = expr;`  (top-level)       | `static const T x = expr;`                         | done   | v0.15                                                |
| `part def P { attribute a : T; … }`          | `typedef struct { … } P;`                          | done   | v0.15 / v0.16                                        |
| `part nested : Other [N]`                    | `Other nested[N];` field                           | done   | v0.17                                                |
| `calc def F { in a:T; in b:T; return:R = e; }` | `R F(T a, T b) { return (e); }`                  | v0.22  | Free function; param order from `in` features        |
| `attribute a : T = expr;`  (in part def)     | line in `P_init(P* self)`: `self->a = expr;`       | v0.23  | Topo-sorted by dependency; cycle is a codegen error  |
| `assert constraint NAME { expr }`            | `if (!(expr)) return false;` in `P_check(P*)`      | v0.24  | All asserts AND'd; named/anonymous → distinct stderr msg |
| `state def S { state a; state b; … }`        | `typedef enum { S_a, S_b, … } S_state;`            | v0.25  |                                                      |
| `transition first X accept E then Y`         | `case X: if (e == S_event_E) return Y; …`          | v0.25  | Falls through default `return s;` (no transition)    |
| `action def A;`  (when used as state event)  | enum member on the owning state def's `S_event`    | v0.25  | Detected via use site, not declaration               |
| `exhibit state s : S`                        | `S_state s;` field on the enclosing struct + init  | v0.25  |                                                      |
| `action def A { first X; then Y; then Z; }`  | `void A(void) { X(); Y(); Z(); }`                  | v0.26  | Sub-action bodies are user-supplied stubs initially  |
| `--emit-c-main` flag                         | skeleton main with init + check + one step         | v0.27  | exit 0 iff every `_check` passed, 1 otherwise        |
| `--emit-c-main` JSON output                  | `<T>_print_json` alongside `<T>_print`             | v0.28  | Selected at runtime via `SML2C_FORMAT=json`          |

Each row is independently shippable.  `make test-c` already pipes
every test file through `cc -fsyntax-only`; the new phases extend
that with `cc -o exe && ./exe` runs against fixture inputs.

## 5. Implementation sequence

### v0.22 — calc defs to C functions

**Why first.**  The expression printer used by attribute defaults
(v0.15) is reusable verbatim; the only new piece is "emit a
function declaration and body."  Lowest-risk, highest-leverage
first step.

**Scope.**
- New `emitCalcDef()` in `src/codegen_c.c`
- Parameters: walk `usage.members` for `direction == DIR_IN`
  features; map their names + types to C parameter list
- Return type: from the `return` feature's type
- Body: `return (<expr>);` where `<expr>` reuses `emitExpr`
- Function name: keep the user's case — `ResponsePower`, not
  `responsePower` or `RESPONSEPOWER`
- Skip-on-uncertainty: any unsupported expression in the body
  causes `/* skipped: NAME (reason) */` on the whole calc def

**Tests.**
- `test/CalcEmit.sysml` with three calc defs (one trivial, one
  with intermediate `attribute scaled = …`, one calling another
  calc)
- `make test-c` runs `cc -fsyntax-only` on the emitted C
- New `make test-c-run`: links the emitted C with a tiny
  hand-written `main` that calls each calc and prints the result;
  fails if output doesn't match expected

**Decision points (commit before coding).**
- Calls to other calc defs in expressions — do they emit `Other(x, y)`
  or `package_Other(x, y)`?  See §6 question 4 below.
- Intermediate `attribute scaled = …` inside a calc body — expand
  inline or emit as a local C variable?  Inline is cleaner unless
  the value is reused; start with local C variables (`double scaled
  = …;` then `return (scaled + 1);`).

### v0.23 — default-value initializers

**Why next.**  The struct typedef from v0.15 leaves all fields
uninitialized.  Today the user has to write `= { .x = 1.0, … }`
inline initializers themselves; with init functions, defaults
declared in SysML actually take effect.

**Scope.**
- For each emitted struct typedef whose part def has at least one
  attribute with a default: emit `void <T>_init(<T>* self)` that
  assigns each defaulted field
- Member references in default expressions (`area = pi*r*r`)
  become `self->pi * self->r * self->r`
- Nested struct fields with their own defaults: chain to inner
  init — `Inner_init(&self->nested);` — *before* assigning
  outer fields that might depend on nested values
- **Initialization order: topological sort by dependency.**  Source
  order would be a sharp edge — the user could write `attribute b =
  a + 1; attribute a = 5;` and trip into reading uninitialized
  memory.  Build a graph (X depends on Y iff X's initializer
  references Y), DFS-with-three-color sort, emit assignments in topo
  order.  A cycle becomes a codegen error that lists every attribute
  in the cycle.  The same machinery applies to top-level non-const
  initialization in the program-init function.

**Tests.**
- A part def with mixed defaults including expressions over
  earlier-declared attributes
- A part def with a nested defaulted part def field
- `make test-c-run`: init, then assert each field equals the
  expected value

### v0.24 — constraints to check functions

**Why next.**  Defaults give us valid initial state; constraints
let us *verify* that state.  Together they make the
"validating library" target real.

**Scope.**
- For each part def with at least one `assert constraint`: emit
  `bool <T>_check(const <T>* self)`
- Each constraint contributes one `if (!(<expr>)) return false;`
- All constraints AND'd implicitly: `_check` returns true iff
  every constraint holds
- `expr` reuses `emitExpr` with `self->X` substitution for
  member references (same machinery as v0.23 init)
- Named constraints: stored in a comment for now (`/* sane_setpoint */`)
  — per-constraint check functions could come later

**Open question: failure detail.**  Three styles:
1. `bool` — caller knows nothing about which constraint failed
2. `int` — return index of first failed constraint, 0 = pass
3. error-list out-parameter — collect all violations

I'd start with (1) for simplicity.  Can extend to (2) later
without breaking the (1) callers.

### v0.25 — state machines

**Why next.**  This is the first lowering that's *not* a pure
function — `Mode_step` reads the current state and returns the
next one.  Once it lands we have the building blocks for a
simulator.

**Scope.**
- `state def S { state a; state b; … }` → `typedef enum { S_a, S_b, … } S_state;`
- Events implied by `accept` clauses: collect every name appearing
  as `accept E` in this state def's transitions, deduplicate, emit
  `typedef enum { S_event_E1, S_event_E2, … } S_event;`
- `transition first X accept E then Y` → `case S_X: if (e == S_event_E) return S_Y; …`
- `transition X then Y` (no accept) → `case S_X: return S_Y;` (unconditional)
- `S_step(S_state, S_event)` returns the new state, or unchanged
  if no transition fires
- `exhibit state mode : S` adds a field `S_state mode;` to the
  enclosing struct, defaulted to the first declared state in init

**Open question: where do entry/exit/do actions go?**  SysML lets
states have `entry F; do G; exit H;` clauses.  Initial answer:
emit entry/exit calls inline in the step function (after a
transition fires, call the source state's `exit` and the target's
`entry`).  Skip `do` for now — it implies continuous activity
which is closer to runtime semantics C than B.

### v0.26 — actions with successions

**Why next.**  An action with `first X; then Y; then Z;` is
sequential code by construction.  Lower it to a function that
calls each named sub-action in order.

**Scope.**
- `action def A { first X; then Y; then Z; }` → `void A(void) { X(); Y(); Z(); }`
- Sub-action declarations (`action X;` inside the action body)
  emit forward declarations as `void X(void);` — the user fills
  in the bodies, or another action def references them
- Named successions with `accept E` clauses are skipped for now —
  those interleave with the state machine, which is more design
  work than the v0.26 budget allows

**Open question.**  Can SysML actions have parameters or local
variables?  Yes (`in` and `out` features).  Initial answer: lower
to function with parameters from `in` features; outputs become
in/out pointer parameters.  But the *body* of an action is a
succession, not an expression — so the lowering is "call each
sub-action in order, threading the outputs through."  This gets
hairy fast; my recommendation is to support parameter-less actions
in v0.26 and defer the full version to v0.28 or later.

### v0.27 — `--emit-c-main` (text output)

**Why last in v1.**  Nice-to-have, not load-bearing for the
library shape.  Generates a minimal `main.c` that picks the first
emittable part def, constructs an instance, calls init, runs all
checks, prints the result.  Useful for "give me a runnable
program from this SysML file" demos and CI integration.

**Scope.**
- Generates `int main(void)` that:
  - calls each emittable type's `<T>_init`
  - calls each `<T>_print` to dump the model state
  - calls each `<T>_check`, capturing the aggregate ok/fail
  - returns `0` iff every check passed, `1` otherwise
- For models with state machines: fires every event once in
  declaration order so the lowering is exercised.  Crude but
  proves the step function works.
- Per-instance: stack-allocated; one of each top-level part def.
  Multi-instance is out of scope.

**Tests.**  `make test-c-run` becomes the headline gate from v0.27
onward.  Per-test `<name>.expect` golden file; the gate compiles
+ runs + diffs stdout.

### v0.28 — JSON output mode

**Why now.**  The text output from v0.27 is good for "did this
run."  Downstream tooling (dashboards, diff tools, programmatic
consumers) wants structured data.

**Scope.**
- Per part def, emit `<T>_print_json(const T*)` alongside
  `<T>_print`
- Generated `main` switches between them via the
  `SML2C_FORMAT=json` environment variable
- String-escape helper for JSON-safe output of `String` fields
- Nested structs print as nested JSON objects; fixed arrays as
  JSON arrays

**Tests.**  Existing test files get parallel `<name>.expect.json`
golden files; `make test-c-run` learns a `--format=json` mode.

## 6. Open design questions

These get settled before the v0.22 turn starts.  I have leanings
but I want explicit confirmation before any code lands.

### Q1: Single file or split header/impl?

**Today.**  `--emit-c` writes one self-contained `.c` file.
Works for the v0.15–v0.17 outputs because there are no function
declarations to share across translation units.

**v0.22+ context.**  Once we have functions, the user wants to
`#include` something to call them from their own code.

**Lean.**  Default emit produces a single `.c` with declarations
and definitions both, so it stays a one-file deliverable.
`--emit-c-header` flag picks header mode (declarations only,
wrapped in `#ifndef` guard).  Users who want both run sml2c
twice with the two flags.

### Q2: Naming convention

**Today.**  Type names preserved verbatim from SysML.  Free
attributes at top level become C globals with their SysML name.

**v0.22+ context.**  Functions get associated with structs:
`Controller_init`, `Controller_check`.  That's `<TypeName>_<verb>`.

**Lean.**  Preserve case throughout.  `Controller`, not `controller`
or `CONTROLLER`.  Less surprising for the user reading both
SysML and emitted C.  C identifiers are case-sensitive so no
collisions from this.

### Q3: Self pointer

**Lean.**  `T*` for mutating ops (init, step), `const T*` for
read-only ops (check).  Standard C library convention.  Allows
multiple instances and chains cleanly.

### Q4: Scoping calc-def names

**Question.**  SysML has hierarchical packages; C does not.  Two
calc defs in different packages can have the same name.  Three
options:

a. **Bare names** — emit `ResponsePower(...)`.  Simple, fails
   loudly on collision (C compiler complains).
b. **Always prefix with package** — emit
   `Thermostat_ResponsePower(...)`.  Verbose but unambiguous.
c. **Mangle on detected collision** — bare names by default,
   prefix when two calc defs collide.

**Lean.**  Start with (a).  PTC has 1580 lines and ~40 calc defs;
collisions in practice are rare enough that the C compiler's
duplicate-symbol error is an acceptable backstop.  Move to (b)
or (c) if this turns out to be a real problem.

### Q5: Constraint failure mode

**Decision (set 2026-04-28).**  `bool` return from `<T>_check`,
plus a stderr message on failure that includes both the type name
and the constraint's source expression:

```c
/* named: `assert constraint inLimits { mass < 500.0 }` */
fprintf(stderr, "Engine.inLimits failed: mass < 500.0\n");

/* anonymous: `assert constraint { torque > 0 }` */
fprintf(stderr, "Engine constraint failed: torque > 0\n");
```

The source expression is always present (it's self-documenting),
the user-given name is prepended when present.  No synthetic
`c0`/`c1` names for anonymous constraints.

When `--emit-c-main` (v0.27) generates a `main`, that `main`
returns 0 iff every `_check` call returned true, 1 otherwise.
Per-constraint detail can come later via a tagged return type
(`int` for index, or struct).

### Q6: Expressions we can't lower yet

Some SysML expressions don't have an obvious C equivalent —
quantity types with units (`75 [kg]`), tuple literals,
collection literals, set comprehensions.  The expression printer
handles a subset of these today (others are flagged as
unsupported).

**Lean.**  Inherit the existing skip-with-comment pattern.  If a
calc def's body or constraint body contains anything `emitExpr`
can't handle, emit `/* skipped: NAME (expression contains
unsupported X) */` on the whole construct.  Don't partial-emit
broken function bodies.

### Q7: Output format for `--emit-c-main`

**Decision (set 2026-04-28).**  Plain indented text in v0.27,
parallel JSON output added in v0.28.

The text format is for "did this run" — a human reads it.  The
JSON format is for downstream tooling (dashboards, diff tools,
CI gates that consume sml2c output programmatically).

The two formats share the data-walking logic.  Per part def,
emit both `<T>_print(const T*)` (text) and `<T>_print_json(const
T*)` (JSON).  The generated `main` switches between them via an
environment variable (`SML2C_FORMAT=json`) or a CLI flag on the
generated binary.

Implementation in v0.28: ~80 lines for the JSON variant of the
print emitter plus a string-escape helper.  Test corpus gets
parallel `<name>.expect.json` files.

## 7. What this is *not*

A few things explicitly outside scope, mentioned so we don't
accidentally drift toward them:

- **No real-time clock.**  No `time_t`, no scheduler.  The user
  drives the system by calling functions.
- **No flows / connections.**  Connecting two ports between part
  defs (`connect a.out to b.in`) is structural in SysML; lowering
  it to C requires choosing a transport (function call?  message
  queue?  shared memory?).  Worth its own design doc when it
  comes up.
- **No code generation for `requirement def` blocks.**
  Requirements compose constraints under named groups; we'd lower
  them as a check-functions-of-check-functions hierarchy.  Easy
  but unsexy; defer.
- **No simulator main loop.**  The user writes `main.c`.  We
  generate the building blocks.

## 8. Test infrastructure additions

`make test-c` today: every `.sysml` test goes through `--emit-c |
cc -fsyntax-only`.  That catches *syntactic* C errors.

For v0.22+ we want *semantic* validation too — emitted code
should compute the right answers.

**New target: `make test-c-run`.**  For each test file with a
companion `<name>.expect` file:

1. Run `bin/sml2c --emit-c <name>.sysml > /tmp/<name>.c`
2. Compile + link with a hand-written `<name>.driver.c` that
   exercises the API
3. Run the resulting executable
4. Diff stdout against `<name>.expect`

A failing diff fails the gate.  Keeps the existing `test-c`
syntactic gate; this one is opt-in per-test.

The thermostat from §2 becomes the first such test.  Calc-emit,
default-emit, constraint-emit, state-emit each get a fixture
in v0.22 / v0.23 / v0.24 / v0.25 respectively.

## 9. Phasing summary

```
v0.22 calc defs       → free functions             (~3 days)
v0.23 init functions  → defaults take effect       (~2 days)
v0.24 check functions → constraints validated      (~2 days)
v0.25 state machines  → enum + step function       (~4 days)
v0.26 action sequence → ordered function calls     (~3 days)
v0.27 main scaffolding (text output, exit 0/1)     (~1 day)
v0.28 JSON output     → parallel _print_json       (~1 day)
```

After v0.27 the thermostat compiles to runnable C end-to-end.
After v0.28 it pipes structured output to downstream tooling.
The PTC reference file (modulo the 15 remaining resolver errors)
becomes meaningfully runnable as a library — its hundred-odd
calc defs become callable functions, its requirements become
check chains, its state machines become enum + step.

That's the destination.

## 10. Decision log

This section accumulates the actual choices we make as v0.22+
turns ship, so the design doc stays in sync with reality.

### 2026-04-28 — pre-v0.22 decisions

- **Q5 (constraint failure):** `bool` return + stderr message of
  the form `"<TypeName>.<name> failed: <expr>"` for named
  constraints, `"<TypeName> constraint failed: <expr>"` for
  anonymous.  No synthetic `c0`/`c1` names.
- **Q7 (output format):** plain indented text in v0.27, parallel
  JSON output in v0.28 via `<T>_print_json` and
  `SML2C_FORMAT=json`.
- **v0.23 init order:** topological sort by initializer
  dependency, with cycle detection as a codegen error.  Source
  order would be a sharp edge.
- **v0.27 main contract:** exit 0 iff every `_check` passed, 1
  otherwise.  For models with state machines, fire every event
  once in declaration order to exercise the lowering.
- **Running example:** thermostat (calc + state machine +
